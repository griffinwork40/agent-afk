---
title: "Skills Overview"
description: "What skills are, how they're scoped, and how to invoke them in agent-afk."
---

Skills are named capabilities that extend what AFK can do in a session. A skill bundles a prompt (or a TypeScript handler) with metadata — name, description, argument hint — and registers itself so the model can call it by name and the REPL can expose it as a slash command.

## What a skill is

When you type `/mint add dark mode` in the REPL, the slash registry resolves `mint` to a skill, injects a dispatch instruction into the session, and the model calls the built-in `skill` tool. The `SkillExecutor` picks up the call, looks up the registered handler, and runs it. All of this happens through one surface: the `skill` tool.

Skills have three pieces of metadata that matter for invocation (defined in `src/skills/index.ts`):

| Field | Purpose |
|---|---|
| `name` | Registry key and slash command name |
| `description` | Shown in `/help` and the skill manifest |
| `argumentHint` | Short usage hint, e.g. `<idea> \| --continue [approved]` |
| `whenToUse` | Routing hint surfaced to the model |
| `flags` | Long-form CLI flags accepted (e.g. `['--continue']`) |

## Scope: built-in, user, and project

Skills come from three origins (`SkillMetadata.origin`, `src/skills/index.ts`):

**Built-in** — TypeScript handlers shipped with AFK, registered at startup via `src/skills/all.ts`. These are the orchestrator skills (`mint`, `diagnose`, `service-setup`, `telegram-setup`). The `origin` field is absent or `'builtin'`.

**User-scope** — `SKILL.md` files discovered under `~/.afk/skills/<skill-name>/`. Scanned lazily at session start by `scanAndRegisterUserSkills()` (`src/skills/user-skills.ts`). The `origin` field is `'user'`.

**Project-scope** — `SKILL.md` files under `<cwd>/.afk/skills/<skill-name>/`. Scanned from the working directory at session start. The `origin` field is `'project'`.

**Plugin skills** — `SKILL.md` files inside installed plugins at `~/.afk/plugins/<plugin>/skills/<skill-name>/`. These are not entered into the TypeScript registry; they route through the plugin skill path in `SkillExecutor`.

When a user or project skill's name collides with an already-registered skill of a different origin, it registers under the namespaced fallback `<origin>:<name>` (e.g. `user:mint`). The earlier registrant keeps the bare name.

## Audience: public vs. internal

Skills can be tagged `audience: 'internal'` (`src/skills/index.ts`). Internal skills are hidden from slash-command listings, `/help`, tab-complete, and the system-prompt skill manifest unless `AFK_INTERNAL=1` is set. They remain dispatchable via `getSkill()` from internal code paths. Most users will only encounter public skills.

## The three execution modes

Every skill has a `context` field that controls how it runs (`src/skills/index.ts`, documented in `docs/skill-load-mode.md`):

| Mode | Mechanic | When to use |
|---|---|---|
| `'inline'` (default) | Calls the TypeScript `handler` in-process | Orchestrators that dispatch sub-agents programmatically (e.g. `mint`, `diagnose`) |
| `'fork'` | Forks a child `AgentSession` with `prompts/system.md` as its system prompt | Heavy or risky work that benefits from an isolated context window (e.g. `service-setup`, `telegram-setup`) |
| `'load'` | Returns `prompts/system.md` / SKILL.md body as the tool result; the **current** agent executes it in-context | Lightweight procedures that should run as the calling agent, with no fork latency |

`load` mode injects an execute-now framing header around the body so the model acts on the instructions rather than summarising them. See `docs/skill-load-mode.md` for full details and authoring guidance.

## How to invoke a skill

**Slash command** — type `/skillname [args]` in the REPL or Telegram:

```
/mint add a dark mode toggle
/diagnose TypeError: Cannot read properties of undefined
/mint --continue approved
```

**`skill` tool** — the model can call the skill directly when it decides to:

```json
{
  "name": "mint",
  "arguments": "add dark mode toggle"
}
```

Both paths route through the same `SkillExecutor`. Slash commands inject a dispatch instruction that causes the model to call the `skill` tool, so the Zod schema flow and structured output are preserved regardless of entry point.

## Invocation telemetry

Every skill dispatch is logged to `~/.afk/agent-framework/routing-decisions.jsonl` (`docs/architecture.md`). You can inspect this file to see which skills ran, when, and with what outcome.
